117 /*
118  * Each bucket may store up to three elements:
119  * - An aligned storage to store a value_type object with placement-new.
120  * - An (optional) hash of the value in the bucket.
121  * - An unsigned integer of type neighborhood_bitmap used to tell us which buckets in the neighborhood of the
122  * current bucket contain a value with a hash belonging to the current bucket.
123  *
124  * For a bucket 'b' a bit 'i' (counting from 0 and from the least significant bit to the most significant)
125  * set to 1 means that the bucket 'b+i' contains a value with a hash belonging to bucket 'b'.
126  * The bits used for that, start from the third least significant bit.
127  *
128  * The least significant bit is set to 1 if there is a value in the bucket storage.
129  * The second least significant bit is set to 1 if there is an overflow. More than NeighborhoodSize values
130  * give the same hash, all overflow values are stored in the m_overflow_elements list of the map.
131  */
132 static const size_t NB_RESERVED_BITS_IN_NEIGHBORHOOD = 2;

1479 /**
1480  * Implementation of a hash map using the hopscotch hashing algorithm.
1481  *
1482  * The Key and the value T must be either nothrow move-constructible, copy-constuctible or both.
1483  *
1484  * The size of the neighborhood (NeighborhoodSize) must be > 0 and <= 62 if StoreHash is false.
1485  * When StoreHash is true, 32-bits of the hash will be stored alongside the neighborhood limiting
1486  * the NeighborhoodSize to <= 30. There is no memory usage difference between
1487  * 'NeighborhoodSize 62; StoreHash false' and 'NeighborhoodSize 30; StoreHash true'.
1488  *
1489  * Storing the hash may improve performance on insert during the rehash process if the hash takes time
1490  * to compute. It may also improve read performance if the KeyEqual function takes time (or incurs a cache-miss).
1491  * If used with simple Hash and KeyEqual it may slow things down.
1492  *
1493  *
1494  * By default the map grows by a factor of 2. This has the advantage to allow us to do fast modulo because
1495  * the number of buckets is kept to a power of two. The growth factor can be changed with std::ratio

1497  * is not a power of two, the map will be slower as it can't use the power of two modulo optimization.

1500  *

1502  * - clear, operator=, reserve, rehash: always invalidate the iterators.
1503  * - insert, emplace, emplace_hint, operator[]: if there is an effective insert, invalidate the iterators
1504  * if a displacement is needed to resolve a collision (which mean that most of the time,
1505  * insert will invalidate the iterators). Or if there is a rehash.
1506  * - erase: iterator on the erased element is the only one which become invalid.
1507  */
